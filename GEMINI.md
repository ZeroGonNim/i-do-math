# I Do Math - Gemini Core Mandates (v1.0)

> **상위 원칙**: 본 파일의 지침은 어떠한 경우에도 최우선 순위(Absolute Precedence)를 갖는다.

---

## 🛑 1. 분석-지시 분리 (Inquiry-First Policy)

- **정의**: 사용자의 모든 요청을 기본적으로 **'Inquiry(분석 및 전략 제안)'**로 간주한다.
- **수행 방식**:
  1.  요청된 사항에 대해 관련 코드와 구조를 분석한다.
  2.  발견된 문제점이나 구현해야 할 구체적인 전략(Strategy)을 **먼저 설명한다.**
  3.  **지시 대기**: "이렇게 수정할까요?"와 같이 확인을 구하고, 사용자가 "수정해라", "작업 시작해라", "OK" 등 명확한 **'Directive(지시)'**를 내릴 때까지 기다린다.
  4.  **수행**: 지시가 확인된 후에만 실제 코드 작성을 시작한다.
- **예외**: "이거 고쳐줘"와 같이 명시적인 명령형이 포함된 경우에만 즉각적인 작업을 허용한다.

## 🛠️ 2. 코드 품질 및 데이터 무결성

- **생략 기호 금지**: 코드 작성 시 `...`와 같은 생략 기호를 절대 사용하지 않는다. 모든 수정은 전체 리터럴 코드를 포함해야 한다.
- **로컬 시간 준수**: 모든 날짜 로직은 브라우저/사용자 로컬 환경 시간을 기준으로 한다. (UTC 오차 방지)
- **실시간 반응형**: 데이터베이스(Dexie/IndexedDB) 변경사항은 항상 `useLiveQuery`를 통해 UI에 즉각 반영되어야 한다.

## 🗺️ 3. 네비게이터 참조 의무

- 모든 작업을 시작하기 전, `docs/PROJECT_NAVIGATOR.md`의 **'1. 현재 개발 상태'**와 **'5. 주요 의사결정 기록'**을 반드시 확인하여 과거의 맥락을 누락하지 않는다.

## 🔢 4. 수학적 검증 원칙 (Verification Principle)

- **코드 기반 검산**: 수학적 정답이나 로직의 오류를 지적하기 전, 반드시 `node -e` 또는 `python3 -c` 명령어로 실제 계산 결과를 확인하라.
- **근거 제시**: "틀렸다"고 보고할 때는 모델의 직관이 아닌, 실행한 코드의 결과값을 근거로 제시하라.
- **철회 기준**: 코드 실행 결과가 데이터의 정답과 일치하면, 지적을 즉시 철회하고 이유를 설명하라.
