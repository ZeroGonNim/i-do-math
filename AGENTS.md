# 🤖 AI 에이전트 행동 지침 (v2.5)

> 세부 코드 규칙은 `rules/`에 있다. 본 문서는 **핵심 프로세스**만 기술한다.

---

## 1. 작업 분류 & Fast-Track

에이전트는 요청 즉시 난이도를 판별하여 대응 속도를 조절한다.

| 등급 | 정의 | 워크플로우 |
|---|---|---|
| **T0 (Fast-Track)** | 단순 오타, 스타일, 상수 변경 | 즉시 실행 후 결과 보고 (계획 생략) |
| **T1 (Standard)** | 기능 추가, 로직 수정 | **[Ask & Plan] 한 번에 제시** → 승인 후 실행 |
| **T2 (Complex)** | 아키텍처 변경, 리팩토링 | 브레인스토밍 → 설계 → 계획 → 승인 → 실행 |

---

## 2. 작업 원칙 (핵심 요약)

1.  **Salami Slicing**: 모든 계획은 2~5분 단위의 작은 태스크로 쪼갠다.
2.  **JIT Reading**: 작업 유형별 필요한 `rules/` 파일만 선택적으로 읽는다.
3.  **Done Criteria**: 작업 시작 전 "완료 기준"을 사용자에게 명확히 전달한다.
4.  **Before/After HOOK**: `rules/hook.md`에 정의된 세션 시작/종료 절차를 수행한다.

---

## 3. WORKFLOW (Turn-Optimized)

### Step 1. [Before HOOK] & [Ask & Plan]
- `rules/hook.md` §1을 실행하여 환경을 진단한다.
- **T1 이상 작업 시**: 모호한 질문(Step 2)과 작업 계획(Step 4)을 **한 턴에 모아서** 사용자에게 제시한다.

```
📋 작업 지시서 & 질문
- 작업 내용: [요약]
- 모호성 질문: [1~2개 핵심 질문]
- 실행 계획 (TDD 기반):
  1. [ ] [태스크 1]
  2. [ ] [태스크 2]
- 완료 기준: [Done Criteria]
```

### Step 2. [Execution] & [After HOOK]
- 승인 후 `rules/hook.md` §2(Working Memory)를 준수하며 작업한다.
- 작업 완료 후 §3(After HOOK)을 실행하여 자가 품질 검증 후 보고한다.

---

## 4. 파괴적 작업 승인 (필수)

다음 작업 전에는 반드시 명시적 승인을 대기한다.
- 패키지 설치/업그레이드, 파일 삭제, 3개 이상 파일의 구조적 리팩토링.
