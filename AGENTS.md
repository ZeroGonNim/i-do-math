# 🤖 Antigravity AI Agent Rules

> 세부 코드 규칙은 `rules/` 디렉토리에 분리되어 있다.
> 이 파일에는 **에이전트의 행동 원칙**과 **작업 프로세스**만 정의한다.

---

## 1. CORE

- 언어/태도: 한국어 필수. 단순 생성기가 아닌 **성능·보안·확장성을 제안하는 기술 파트너.**
- Tech Stack 및 절대 금지 사항: `CLAUDE.md` §3, §4 참조.
- 코드 규칙: 작업 유형에 따라 `CLAUDE.md` §6 라우팅 테이블에서 해당 `rules/` 파일을 참조.

---

## 2. 출력 포맷

작업 복잡도에 따라 유연하게 조절한다.

- **단순 수정/버그픽스**: 코드 + 핵심 변경 사유 한 줄. 불필요한 설명 생략.
- **신규 기능/구조 설계**: 문제 분석 → 해결 전략(선택지 비교) → 코드

---

## 3. 작업 원칙

1. **문서 우선 참조**: 작업 전 반드시 `docs/` 폴더의 문서나 `Implementation Plan`을 먼저 읽고 컨텍스트를 파악한다.
   - **와이어프레임 작업 시**: `docs/wireframe/wireframe-rules.md`(인덱스)를 먼저 읽고, 작업 단계에 따라 하위 파일을 참조한다.
2. **단계별 작업**: 한 번에 하나의 컴포넌트/기능만 처리하여 품질을 높인다.
3. **컨텍스트 리셋 (대규모 작업 시)**: 작업량이 많아 컨텍스트가 누적되면 품질이 저하된다. 다음 기준으로 리셋한다:
   - **코드**: 3개 이상 파일에 걸친 기능 구현 시, 파일 단위로 나눠 작업하고 중간 결과를 커밋/파일로 기록한 뒤 다음 단계를 시작한다.
   - **와이어프레임**: 1회 실행 = 1화면(4디바이스). 다음 화면은 이전 결과를 확인한 후 새로 시작한다.
   - **문서/기획**: 대규모 문서 작성 시 섹션 단위로 분할하여 각 섹션을 독립적으로 완성한다.
4. **완료 기준 사전 합의**: 작업 시작 전 "어떤 조건을 충족하면 완료인지(Done Criteria)"를 사용자와 합의한다. 검수 시 이 기준으로 판단한다.
5. **실행 가능 코드**: 항상 실행 가능한 구조로 제공하며, 필요한 `yarn` 명령어를 포함한다.
6. **개발 우선, 운영 후행 (철칙)**: 모든 변경은 먼저 로컬(Development) 환경에서 검증한다. 운영(Production) 배포인 `npx vercel --prod`는 사용자가 로컬 결과를 확인한 후 **명시적 승인("운영 배포해줘")을 할 때만** 진행한다.

---

## 4. 필수 승인 (Plan Approval)

다음 **파괴적 작업** 수행 전에는 반드시 사용자에게 이유를 설명하고 승인을 대기한다:

- 새로운 패키지 설치 또는 메이저 버전 업그레이드
- 기존 파일 삭제 및 폴더 구조 변경
- 대규모 리팩토링 (3개 이상 파일에 걸친 구조 변경)
- 공용 컴포넌트/훅의 인터페이스 변경 (breaking change)
- 환경 변수 추가/삭제

---

## 5. WORKFLOW

```
요청 접수 → Planner(해석·분류) → 작업 지시서 확인 → 작업 수행 → 경량 검수 → (리뷰 요청 시) FM 검수
```

### 5-1. Planner 프로세스 (요청 해석 → 작업 지시서)

사용자의 자연어 입력을 **있는 그대로 실행하지 않는다.** 먼저 아래 단계를 거쳐 작업 지시서로 변환한다.

#### Step 1. 작업 유형 분류
`CLAUDE.md` §6 라우팅 테이블을 기준으로 작업 유형을 판별한다.
복합 유형(예: "로그인 페이지 만들어줘" → 컴포넌트 + API 연동 + 상태 관리)인 경우 모두 식별한다.

#### Step 2. 모호성 확인 → 질문
요청에 다음 항목이 명시되지 않았으면 **작업 전에 사용자에게 질문**한다:

| 확인 항목 | 질문 예시 |
|---------|---------|
| 범위(Scope) | "소셜 로그인도 포함하나요, 이메일/비밀번호만?" |
| 데이터 흐름 | "API는 REST? GraphQL? 기존 엔드포인트가 있나요?" |
| UI 기대치 | "기존 디자인 시안이 있나요, 아니면 자유 구성?" |
| 우선순위 | "MVP 수준? 아니면 에러 처리·로딩 상태까지?" |
| 의존성 | "기존 인증 스토어나 API 클라이언트가 있나요?" |

> 단순 작업(버그 수정, 텍스트 변경 등)은 질문 없이 바로 진행해도 된다.

#### Step 3. 참조 규칙 결정
작업 유형에 따라 `rules/` 파일과 `docs/` 문서를 결정한다.

#### Step 4. 작업 지시서 생성 → 확인
위 정보를 종합하여 사용자에게 요약 제시 후 승인을 받는다:

```
📋 작업 지시서
- 작업: [구체적 작업 내용]
- 범위: [포함/제외 항목]
- 참조 규칙: [rules/파일, docs/파일]
- 완료 기준: [Done Criteria]
- 예상 산출물: [파일명, 컴포넌트명 등]
```

> **단순 작업**은 작업 지시서 없이 바로 실행.
> **복합 작업**(2개 이상 파일, 신규 기능)은 반드시 작업 지시서를 먼저 제시.

### 5-2. 작업 수행 → 검수

작업 지시서 승인 후:
1. 해당 `rules/` 파일 참조하여 작업 수행
2. §3 작업 원칙 준수 (컨텍스트 리셋, 완료 기준 등)
3. 작업 완료 → §6-1 경량 검수 자동 수행
4. 사용자 요청 시 → §6-2 FM 검수 실행

---

## 6. 검수 체계 (2단계)

### 6-1. 경량 검수 (자동 — 모든 작업 완료 후 필수)

모든 작업 완료 후 에이전트가 자체적으로 수행한다. 이슈 발견 시 즉시 수정 후 결과를 보고한다.

> ⚠️ **자기평가 편향 방지**: 경량 검수 시 "내가 잘 만들었는지"가 아닌, "체크리스트의 각 항목이 객관적으로 충족되었는지"만 판단한다. 작업 중 세운 가정이나 의도를 배제하고, 결과물만 보고 판단할 것.

| 작업 유형 | 체크 항목 |
|---------|---------|
| 와이어프레임 | `docs/wireframe/wireframe-checklist.md` 전체 수행 |
| 코드 | 타입 에러 없음, 빌드 통과, CLAUDE.md §3 절대 금지 사항 준수 |
| 문서 | 필수 섹션 존재 여부, 기존 구조와의 정합성 |

### 6-2. FM 검수 (수동 — 사용자 요청 시)

사용자가 "검수해줘", "리뷰해줘" 등 명시적으로 요청할 때 실행한다.
4개 파트가 병렬 검수하되 **이슈가 있는 파트만 출력**한다.

| 파트 | 책임 |
|---|---|
| 기획 | 요구사항 검증, 비즈니스 로직 정합성 |
| 디자인 | UX 흐름, 접근성(WCAG 2.1 AA), 반응형 |
| 개발 | 아키텍처 적합성, 성능 최적화 |
| QA | 엣지 케이스, 크로스 브라우저 호환성 |

복잡한 작업(구조 변경, 신규 기능)에서 사용자가 요청 시 멀티 라운드(독립→교차→합의) 검수를 진행한다.
